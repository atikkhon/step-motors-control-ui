# Step Motors Control UI

**Step Motors Control UI** — это desktop-приложение на **C++ / Qt Widgets** для операторского управления группой шаговых приводов через **Modbus TCP**. Проект реализует HMI-панель, которая подключается к PLC или Modbus TCP simulator, циклически считывает состояния системы и приводов, а также отправляет команды управления в карту holding-регистров.

**Цель проекта** — реализовать практическую HMI-систему для управления группой шаговых приводов, приближенную к задачам промышленной автоматизации: обмен с PLC/контроллером по Modbus TCP, работа с регистровой моделью данных, выдача команд управления, чтение статусов и диагностических сигналов, контроль связи через heartbeat и удобное отображение состояния оборудования для оператора.

## Что делает приложение

**Step Motors Control UI** работает как HMI для системы из **22 шаговых моторов**. Приложение позволяет оператору подключиться к Modbus TCP устройству, контролировать общее состояние установки, задавать скорость движения и управлять каждым приводом отдельно.

Основные возможности:

- подключение и отключение от PLC / Modbus TCP server;
- отображение текущего статуса соединения;
- циклический опрос holding-регистров с периодом 500 мс;
- отправка команд только при изменении управляющих данных;
- контроль общего состояния системы: готовность, движение, ошибка, аварийный останов;
- формирование heartbeat-сигнала для контроля активности HMI;
- глобальная команда остановки системы;
- отдельная активация верхней и нижней групп приводов;
- задание общей скорости движения приводов;
- управление 22 моторами из единого интерфейса;
- для каждого мотора доступны команды `Jog +`, `Jog -`, движение в точку, homing, stop и acknowledge;
- отображение статусов каждого мотора: активирован, занят, ошибка, остановлен, выполняет jog, выполняет homing, едет в точку, был обнулен;
- отображение текущей позиции мотора;
- ввод уставки положения с валидацией числового значения;
- разделение логики Modbus-обмена, модели устройств и графического интерфейса.

## Архитектура

Проект построен по схеме **Qt Widgets UI + C++ backend + Modbus TCP communication layer**.

UI-часть отвечает за операторские элементы управления и индикацию. C++ backend описывает абстракции устройств, хранит write/read-регистры, преобразует команды в биты и слова Modbus, а также обновляет интерфейс при поступлении новых данных от PLC.


| Файл | Роль |
|---|---|
| `main.cpp` | Точка входа в приложение. Создает `QApplication`, главное окно `MainWindow` и запускает event loop Qt. |
| `mainwindow.h / mainwindow.cpp` | Собирает всю HMI-панель: создает 22 объекта моторов, системный объект, объект скорости и Modbus-коммуникатор. Подключает устройства к слою обмена и распределяет виджеты моторов по двум scroll-областям. |
| `mbcomitem.h / mbcomitem.cpp` | Описывает абстрактный интерфейс Modbus-устройства `MbComItem` и конкретные модели данных: `MbComMotor`, `MbComSystem`, `MbComSpeed`. Здесь реализованы write/read-регистры, битовые команды, статусы и масштабирование позиции. |
| `modbuscommunicator.h / modbuscommunicator.cpp` | Слой связи с PLC. Использует `QModbusTcpClient`, формирует read/write-запросы к holding-регистрам, выполняет polling по таймеру, разбирает ответы и передает данные в объекты устройств. |
| `modbusview.h / modbusview.cpp / modbusview.ui` | Виджет подключения к PLC: кнопка connect/disconnect и индикация состояния Modbus-соединения. |
| `systemview.h / systemview.cpp / systemview.ui` | Виджет системного управления: heartbeat, stop, активация верхней и нижней группы приводов, индикация состояния системы. |
| `speedview.h / speedview.cpp / speedview.ui` | Виджет задания общей скорости. Использует `QIntValidator` и передает уставку в `MbComSpeed`. |
| `motorview.h / motorview.cpp / motorview.ui` | Виджет одного мотора. Содержит команды движения, остановки, квитирования, homing, задание позиции и отображение статусов. Один и тот же компонент используется для всех 22 приводов. |

## Как реализовано

### Modbus TCP обмен

За связь с контроллером отвечает класс `ModbusCommunicator`. Он инкапсулирует работу с `QModbusTcpClient` и скрывает детали Modbus-обмена от UI-компонентов.

Опрос выполняется по таймеру с периодом **500 мс**. При активном соединении приложение отправляет read request к holding-регистрам, получает блок данных и распределяет его между объектами системы и моторов.

Write request формируется только при изменении данных в одном из устройств. Для этого у объектов `MbComSystem`, `MbComSpeed` и `MbComMotor` есть флаг обновления данных. Такой подход применяется, так как интерфейс не должен постоянно перезаписывать управляющие регистры без необходимости.

### Карта регистров

В проекте используется раздельная область для чтения статусов и записи команд.

| Область | Смещение | Назначение |
|---|---:|---|
| Read registers | `0` | Чтение статусов системы и 22 моторов |
| Write registers | `120` | Запись команд системы, скорости и команд моторов |

Read-блок состоит из:

| Устройство | Количество регистров | Адреса |
|---|---:|---|
| System | 1 | `0` |
| Motor 1 | 5 | `1–5` |
| Motor 2 | 5 | `6–10` |
| ... | ... | ... |
| Motor 22 | 5 | `106–110` |

Write-блок состоит из:

| Устройство | Количество регистров | Адреса |
|---|---:|---|
| System commands | 1 | `120` |
| Speed setpoint | 1 | `121` |
| Motor 1 commands | 5 | `122–126` |
| Motor 2 commands | 5 | `127–131` |
| ... | ... | ... |
| Motor 22 commands | 5 | `227–231` |

Такая структура делает систему удобной для сопровождения, так как логика HMI напрямую связана с картой регистров контроллера, поэтому данные проще проверяются при пусконаладке и легче масштабируются.

### Системные команды и статусы

Системный объект `MbComSystem` использует один write-регистр и один read-регистр.

Write-регистр системы:

| Бит | Назначение |
|---:|---|
| `0` | Heartbeat |
| `1` | Stop system |
| `2` | Activate upper motors group |
| `3` | Activate lower motors group |

Read-регистр системы:

| Бит | Назначение |
|---:|---|
| `0` | System ready |
| `1` | System moving |
| `2` | System error |
| `3` | Emergency stop |

Heartbeat переключается по таймеру каждые 500 мс - контроль связи между HMI и контроллером.

### Управление мотором

Каждый мотор представлен объектом `MbComMotor`. Для одного мотора используется **5 write-регистров** и **5 read-регистров**.

Основной write-регистр содержит битовые команды:

| Бит | Команда |
|---:|---|
| `0` | Activate |
| `1` | Jog forward |
| `2` | Jog backward |
| `3` | Go home |
| `4` | Go point-to-point |
| `5` | Stop |
| `6` | Acknowledge |

Основной read-регистр содержит статусы:

| Бит | Статус |
|---:|---|
| `0` | Activated |
| `1` | Busy |
| `2` | Jog forward active |
| `3` | Jog backward active |
| `4` | Homing active |
| `5` | Point-to-point active |
| `6` | Alarm |
| `7` | Homed |
| `8` | Stopped |

Позиция передается как 32-битное signed-значение через два 16-битных регистра. В приложении используется масштабирование `×1000`, то есть значение `10.500` в интерфейсе преобразуется в `10500` для передачи в регистры. При чтении текущей позиции выполняется обратное преобразование.

### Импульсные команды

Часть команд реализована как импульс с заданой длительностью.

После нажатия кнопки соответствующий бит выставляется в `true`, а через заданный период времени автоматически сбрасывается обратно. Это практически обеспечивает, что при установленной связи с ПЛК команда будет доставлена, и ПЛК заметит изменение фронта команды.

### Jog-режим

Команды `Jog +` и `Jog -` работают по принципу удержания кнопки:

- при `pressed()` бит команды устанавливается;
- при `released()` бит команды сбрасывается.

Мотор движется только пока оператор удерживает кнопку.

### Обновление интерфейса

После каждого успешного чтения Modbus-регистров данные передаются в соответствующие объекты устройств. Затем объекты генерируют сигнал `stateUpdated()`, а виджеты обновляют чекбоксы, текстовые поля и текущую позицию.

Например, `MotorView` не читает Modbus напрямую. Он работает только с объектом `MbComMotor`, а вся низкоуровневая связь вынесена в `ModbusCommunicator`. Такое разделение упрощает поддержку проекта и позволяет переиспользовать один и тот же виджет для любого количества приводов.

## Какой опыт получен

Был получен опыт, близкий к задачам разработки HMI/SCADA и интерфейсной части АСУ ТП:

- разработка операторского интерфейса для управления исполнительными механизмами;
- организация обмена с PLC Modbus TCP server;
- работа с промышленным протоколом **Modbus TCP** через Qt Serial Bus;
- проектирование карты регистров для команд и статусов;
- разделение read-регистров и write-регистров;
- упаковка дискретных команд и статусов в отдельные биты 16-битных слов;
- преобразование величин в регистровое представление и обратно;
- реализация циключеского опроса для обновления состояний оборудования;
- реализация heartbeat-сигнала между HMI и контроллером;
- обработка состояний соединения: подключение, отключение, промежуточные состояния;
- построение масштабируемого UI для большого количества однотипных приводов;
- разделение глобального управления системой и локального управления каждым мотором;
- реализация импульсных команд для PLC-логики;
- реализация jog-управления через `pressed/released` события;
- отображение диагностики приводов: busy, alarm, stopped, homed, active.

Также был получен практический C++ / Qt опыт:

- разработка desktop-приложения на **C++ / Qt Widgets**;
- работа с `QObject`, signals/slots и event loop Qt;
- использование `QTimer` для циклических и импульсных операций;
- работа с `QVector`, `QString`, `QModbusDataUnit`, `QModbusReply`;
- создание переиспользуемых виджетов;
- работа с Qt Designer и `.ui`-формами;
- использование `QIntValidator` и `QDoubleValidator` для проверки вводимых уставок;
- сборка проекта через qmake.
